## 二、系统架构与数据流（约4分钟）

### 2.1 整体架构

系统采用标准的**前后端分离架构**，有三层：

```
前端（Vue 3 + Element Plus）
        ↕ REST API（axios）+ WebSocket（Socket.IO）
后端（Flask + Flask-SocketIO + eventlet）
        ↕ SQLAlchemy ORM
数据库（SQLite / MySQL）   +   AI模块（PyTorch + MTCNN）
```

前端是一个 Vue 3 的单页应用，有 11 个功能页面；后端用 Flask 的工厂模式 + Blueprint 分模块，共 7 个 Blueprint、约 30 个 REST 接口；AI 模块独立于 Web 层，以懒加载方式在第一次推理请求时初始化，训练完成后可热更新，无需重启服务。

**四种角色权限边界**（通过 JWT Payload 的 role 字段控制）：
- 管理员（admin）：全权限——用户管理、学生管理、触发训练
- 教师（teacher）：创建并主持考勤任务、查看报表、纠正待核实记录
- 辅导员（counselor）：查看报表和缺勤预警，无写权限
- 学生（student）：仅查看自己的出勤记录

### 2.2 核心数据流：一帧视频到考勤记录

这是整个系统最核心的数据通路，完整链路如下：

```
教师浏览器
  ├─ 每500ms  canvas.toDataURL('image/jpeg', 0.7)
  └─ socket.emit('frame', {task_id, image: base64_data_uri})
        ↓ WebSocket（Socket.IO room = task_{id}）
后端 ws/handlers.py  handle_frame()
  ├─ 1. 剥离 data:image/jpeg;base64, 前缀
  ├─ 2. base64 解码 → numpy uint8 数组（BGR格式）
  └─ 3. 调用 AttendanceService.process_frame(task_id, frame)
              ↓
        AI推理链路（backend/app/ai/）
        ┌─────────────────────────────────────────┐
        │  FaceDetector（facenet_pytorch MTCNN）   │
        │    BGR→RGB → PIL Image                   │
        │    三级联检测（P-Net→R-Net→O-Net）        │
        │    输出：bboxes + landmarks + confidence │
        │    过滤 prob < 0.7 的候选框              │
        ├─────────────────────────────────────────┤
        │  Preprocessor                            │
        │    按 bbox 裁剪（外扩20px padding）       │
        │    resize → 224×224                      │
        │    Normalize（ImageNet均值/标准差）        │
        ├─────────────────────────────────────────┤
        │  Embedder（ResNet50 截断到 fc[0:2]）     │
        │    前向传播 → 512-dim 特征向量            │
        │    L2 归一化                              │
        ├─────────────────────────────────────────┤
        │  FeatureDatabase（features.npy）         │
        │    cosine_similarity = feat @ db.T        │
        │    sim > 0.75 → present（出勤）           │
        │    0.50 < sim ≤ 0.75 → unverified（待核实）│
        │    sim ≤ 0.50 → unknown（忽略）           │
        └─────────────────────────────────────────┘
              ↓
        AttendanceRecord 写库（去重：task_id + student_id 唯一约束）
              ↓
        socketio.emit('recognition_result', results, room=f'task_{task_id}')
              ↓ 广播到同一考勤任务的所有客户端
教师浏览器实时更新出勤名单
```

整条链路的 CPU 端到端延迟约 **176ms**（MTCNN ~60ms + ResNet50 ~100ms + 匹配 ~16ms），在 500ms 发帧间隔下有约 300ms 的余量，满足实时性要求。

### 2.3 双阈值机制的设计考量

为什么要引入"待核实"中间状态，而不是简单地用一个阈值？

在实际场景中，拒识（FRR）和误识（FAR）之间存在不可避免的权衡。单阈值方案非此即彼：调高阈值降低误识但拒识增加，调低又反之。引入第二个阈值（0.50）后，相似度在 0.50~0.75 之间的"模棱两可"情况不自动判定，而是留给老师人工复核。这样：
- 高于 0.75 → 高置信度，系统自动记出勤，教师无需干预
- 0.50~0.75 → 中等置信度，标为待核实，保留人工纜底
- 低于 0.50 → 低置信度，视为非注册学生直接忽略，避免乱写记录

从实验数据看，ResNet50 的 FAR 只有 1.17%，说明这套机制有效地把误识控制在了极低水平。

## 三、AI模块设计与实验（约5分钟）

### 3.1 数据集

使用 LFW（Labeled Faces in the Wild）数据集的 Top-50 子集（图片数最多的50人），按 70/20/10 比例划分三个不重叠子集：

| 子集 | 样本数 | 每类均值 |
|------|--------|---------|
| 训练集 | 1,920 | 38.4 张 |
| 验证集 | 533 | 10.7 张 |
| 测试集 | 320 | 6.4 张（封存）|

选择 50 类是有考虑的：实际课堂一般 30~60 人，50 人是一个合理的模拟规模。测试集在整个实验期间完全封存，只在最终评估时使用一次，保证评估结果的可信度。

**类别不平衡**：最多类（George_W_Bush，371张）是最少类的近14倍。为此，训练时采用了**加权随机采样（WeightedRandomSampler）**和**加权交叉熵损失**，让少样本类在训练中获得更多关注。

### 3.2 数据增强策略

针对考勤场景的实际变化，增强策略涵盖以下几类：

| 增强操作 | 参数 | 模拟场景 |
|---------|------|---------|
| ColorJitter | brightness/contrast=0.4, saturation=0.3 | 教室灯光变化 |
| RandomGrayscale | p=0.1 | 低照度/部分灰度摄像头 |
| RandomAffine | degrees=±20, translate=0.1, scale=0.9~1.1 | 低头/仰头/位移 |
| GaussianBlur | kernel=3, sigma=0.1~2.0 | 摄像头模糊/运动模糊 |
| RandomErasing | p=0.2, scale=2%~15% | 遮挡（刘海、眼镜、口罩） |
| RandomHorizontalFlip | — | 侧脸变化 |

### 3.3 两阶段迁移学习策略

**为什么用迁移学习？** 每类学生只有约38张训练图片，从头训练几乎不可能收敛，迁移学习是在小样本场景下几乎唯一可行的路线。

两阶段设计：

```
Phase 1（Epoch 1~10）：冻结 ResNet50 卷积主干
  ├─ 只训练新增分类头（2048→512→C，约107万参数）
  ├─ 优化器：Adam，lr=1e-3（固定，无衰减）
  └─ 目的：让随机初始化的分类头在 ImageNet 特征上快速收敛
        → Epoch 10 验证集准确率 45.40%

Phase 2（Epoch 11~30）：解冻全部参数
  ├─ 优化器：SGD + Momentum（0.9），weight_decay=1e-4
  ├─ 调度器：CosineAnnealingLR（T_max=20，eta_min=1e-6）
  └─ 目的：在已收敛分类头的基础上精细调整全部参数
        → Epoch 26 达峰值 86.30%（最优检查点）
```

Phase 1 用 Adam 是因为它对学习率不敏感，收敛快；Phase 2 换 SGD 是因为在微调任务上 SGD + 余弦退火有更好的泛化性能，且内存占用约是 Adam 的三分之一，在 CPU 服务器上更友好。

### 3.4 VGG16 vs ResNet50 对比实验

这是本文最重要的实验结论之一。两个模型的核心差异：

| 维度 | ResNet50 | VGG16-GAP |
|------|---------|----------|
| 卷积主干参数量 | 约2550万 | 约1.38亿（5.4倍） |
| 分类头 | FC(2048→512→C) | GAP→FC(512→512→C) |
| batch_size | 16 | 32（GAP改造后） |
| Phase1末 val_acc | **45.40%** | 43.34% |
| 最优 val_acc | **86.30%**（Ep.26） | 74.67%（Ep.26） |
| 测试集 Top-1 | **81.87%** | 71.25% |
| AUC-ROC | **99.30%** | 98.22% |
| FAR | 1.17% | **0.99%** |
| FRR | **18.12%** | 28.75% |
| CPU 推理延迟 | **176ms** | 405ms |

VGG16 原本用三层全连接头（约1.08亿参数），极大的参数量在 batch_size=4 下梯度噪声太大，几乎无法收敛。改为 GAP 架构后，分类头参数从1.08亿降至26万，batch_size 可以提到32，精度大幅提升。但推理延迟的瓶颈在特征提取而非分类头，GAP 对延迟几乎没有帮助——405ms 是 176ms 的 2.3 倍，超目标一倍，这是主干参数量带来的结构性约束。

**结论：ResNet50 是本系统 CPU 部署场景下唯一可行的选择。**

## 四、关键工程实现（约3分钟）

### 4.1 WebSocket 实时推送

考勤的实时性要求前端能看到"刚刚某人被识别到了"，而不是等任务结束后刷新。传统 REST 轮询每次需要完整的 HTTP 握手，延迟高且浪费资源。

本系统用 Flask-SocketIO 实现双向通信：
- 前端用 `socket.emit('frame', ...)` 推送视频帧
- 后端用 `socketio.emit('recognition_result', ..., room=f'task_{id}')` 按任务隔离广播

房间（Room）机制确保不同班级的考勤结果互不干扰。实测两个考勤任务同时运行时，单帧延迟差小于 20ms。

### 4.2 特征库热加载机制

训练完成后，模型需要更新，但不能重启正在运行的考勤任务。解决方案：
- 特征库存储在 `features.npy`，key=student_id，value=512维向量
- `ModelManager` 持有读写锁（`threading.RLock`）
- 训练完成后在锁内调用 `reload_recognizer()`，原子性替换 `_recognizer` 实例
- 正在处理的帧用旧实例完成，下一帧自然切换到新实例

### 4.3 多角色权限设计

用装饰器模式实现三种权限控制粒度：
```python
@require_auth           # 仅验证 JWT 有效
@require_roles("admin") # 验证 JWT + 角色白名单
@require_roles("admin", "teacher")  # 多角色允许
```

学生"只能查自己记录"的逻辑：通过 JWT 中的 user_id 反查 Student 表获得 student.id，再与路由参数 student_id 比对，避免了直接比较不同表主键的错误。

### 4.4 考勤记录去重

同一学生在一次考勤中可能被识别多次（500ms 发帧，任务可能持续30分钟），需要防止重复写库。两层防护：
1. 进程内存缓存 `{task_id: set(student_db_id)}`，快速过滤重复
2. 数据库 `attendance_records` 表的唯一约束 `(task_id, student_id)`，做兜底保障

## 六、不足与展望（约1分钟）

诚实说，这个系统有几个明确的局限性：

**当前不足：**
1. **精度指标部分未达标**：测试集 Top-1（81.87%）、F1（79.16%）低于目标 85%，根本原因是类别不平衡（最多类样本是最少类的14倍）。已引入加权采样和加权损失，有望在重训练后改善。
2. **推理延迟接近上限**：CPU 单帧 176ms，余量不大，若同时有多个考勤任务并发，可能出现排队现象。
3. **训练在 Web 进程内用 Thread 执行**：约90分钟的训练会占满 GIL，影响同期接口响应，生产环境应改为 Celery 异步任务。

**改进方向：**
- 近期：用加权训练策略重训，预期 macro F1 提升到 78% 以上
- 中期：INT8 量化（`torch.quantization.quantize_dynamic`），推理延迟可压到 60~80ms
- 长期：将 Softmax 分类损失换成 ArcFace 损失，人脸特征的可分性会显著提升，预期准确率超过 90%

## 七、结语

总结一下本项目完成的工作：设计并实现了前后端分离的全功能考勤系统，实现了 MTCNN + ResNet50 的实时人脸识别推理链路，进行了 VGG16 和 ResNet50 的系统性对比实验，为 CPU 资源受限场景下的模型选型提供了量化依据。

主要结论：
- ResNet50 在准确率、速度、资源占用三个维度综合优于 VGG16
- 双阈值机制能有效平衡误识率和拒识率，比单阈值方案更实用
- 两阶段迁移学习在小样本场景下是可行且高效的训练策略

## 附：常见答辩问题预案

**Q1：为什么选 ResNet50 而不是更新的 EfficientNet 或 MobileNet？**

> 主要考虑三点：①任务书明确要求比较 VGG16 和 ResNet50 这两种架构；②ResNet50 在 LFW 等人脸基准上有大量公开对比数据可参考；③MobileNet 虽然更轻量，但在本实验的小样本场景（每类约38张）下，特征提取能力的差异会放大，ResNet50 的 2500 万参数相对于数据集来说已经足够，再轻量化收益有限。

**Q2：余弦相似度 vs 欧氏距离，为什么选余弦？**

> 我们对特征向量做了 L2 归一化，归一化后欧氏距离和余弦相似度是等价的（差了一个常数变换）。实际上两者都可以，选余弦是因为它的输出在 [-1,1] 范围内，对阈值的意义更直观，0.75 这个数字解释起来更自然。

**Q3：双阈值（0.75 / 0.50）是怎么确定的？**

> 是通过在验证集上枚举阈值组合确定的。以 0.75 为高阈值时，FAR 约 1.2%，误识率可接受；低于 0.50 时大量样本与任何注册学生都相差甚远，基本可以认定为非注册人员。这两个值不是魔法数字，可以根据实际部署场景调整——如果对误识容忍度更低，可以把高阈值提到 0.80。

**Q4：类别不平衡问题的影响有多大？**

> 影响显著。ResNet50 的加权平均 F1 是 79.16%，但宏平均 F1（每类等权）只有 72.16%，差了约 7 个百分点，这个差距就是类别不平衡造成的。少样本类的 FRR 高达 26.74%，意味着每4次识别有1次会漏识，这对实际使用影响很大。本文已经引入加权采样和加权损失，如果时间允许完整重训，预期能改善这个问题。

**Q5：MTCNN 的三级联检测具体是怎么工作的？**

> MTCNN 由三个级联的 CNN 组成：P-Net（Proposal Network）生成候选区域，R-Net（Refine Network）去掉大部分假阳性，O-Net（Output Network）输出精确的 bbox 和 5 个关键点（双眼、鼻尖、双口角）。三级处理从粗到细，在保证精度的同时控制计算量。关键点用于人脸对齐，把检测到的人脸旋转矫正到正面，减少后续特征提取时的角度干扰。

**Q6：WebSocket 和 REST 各自用在哪里，怎么划分的？**

> 划分标准很简单：需要服务端主动推送的用 WebSocket，请求-响应式的用 REST。具体来说：视频帧上传和识别结果推送用 WebSocket，因为老师需要实时看到谁来了，服务端必须主动通知；创建任务、查询记录、导出报表等都是请求-响应式操作，用 REST 更合适。混合使用的好处是各取所长，不会因为非必要的 WebSocket 连接增加服务端负担。

**Q7：系统目前最大的安全隐患是什么？**

> 最主要的是禁用账号不能即时失效 JWT 的问题。JWT 是无状态的，签发后有效期24小时，管理员在后台禁用账号后，该用户持有的 token 在有效期内仍然可以访问所有 API。解决方案是引入 JWT 黑名单（用 Redis 存被吊销的 jti），在每次请求时查询黑名单。当前版本因为没有引入 Redis，暂时没有实现这个功能，这是已知的技术债，实际部署前需要补上。

**Q8：如果同时有 100 个学生的摄像头接入，系统能撑住吗？**

> 当前架构撑不住。限制来自两方面：①Flask + eventlet 单进程模式，AI 推理是 CPU 密集任务，100 个并发帧会排队，延迟会线性上升；② Python GIL 在多线程 CPU 密集场景下实际上是单核运行。改进方案是把 AI 推理抽离为独立的 Celery worker（可多进程），Web 层只做帧接收和结果转发，这样水平扩展才现实。当前版本的定位是班级级别（单任务30~60人，1fps），在这个规模下运行没有问题。
